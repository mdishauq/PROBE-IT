Overview

Core Platform & Design Philosophy

Architecture Overview

Probe IT operates as two cooperating subsystems working seamlessly together:

  • Native Engine (C++): Samples OS counters and computes derived metrics
  • Flutter Frontend (Dart): Receives live telemetry via platform channels
  • Data Flow: C++ → Plugin Layer → Flutter UI
  • Concurrency: Continuous sampling without blocking UI work

Design Goals

Built with precision and efficiency:

  • Accurate native readings
  • Low overhead operation
  • Safe concurrency
  • Simple serialization

Key Constraints

Platform support and reliability:

  • Cross-platform portability
  • Portable sampling code

Windows Integration

Native API usage for precision:

#ifndef HEALTH_SENSOR_HPP
#define HEALTH_SENSOR_HPP
#include <Windows.h>
#include <stdint.h>
#include <cmath>

enum class DataUnit : uint8_t {
    Bytes, KB, MB, GB, TB
};

class health_sensor {
public:
    struct health_reads {
        uint8_t MemoryLoad;
        uint64_t TotalRam;
        uint64_t RamAvailable;
        uint64_t TotalPageFile;
        uint64_t TotalAvailPageFile;
        uint64_t TotalVirtualMemory;
        uint64_t TotalAvailVirtualMemory;
    };

    struct ResultPackage {
        double Value;
        DataUnit unit;
    };

    uint64_t prevIdle, prevKernel, prevUser;
    
    void CPU_AttributesRetriver(void);
    void UsedRamCalculator(ResultPackage* rstptr);
    double RamUsageLevel();
    double CpuWorkingLoad();
};

#endif

Memory Sampling (health_sensor.cpp)

void health_sensor::CPU_AttributesRetriver(){
    winState.dwLength = sizeof(winState);
    if(GlobalMemoryStatusEx(&winState)){
        // Retrieve system memory statistics
        health_data.MemoryLoad = winState.dwMemoryLoad;
        health_data.TotalRam = winState.ullTotalPhys;
        health_data.RamAvailable = winState.ullAvailPhys;
        health_data.TotalPageFile = winState.ullTotalPageFile;
        health_data.TotalAvailPageFile = winState.ullAvailPageFile;
        health_data.TotalVirtualMemory = winState.ullTotalVirtual;
        health_data.TotalAvailVirtualMemory = winState.ullAvailVirtual;
    }
}

void health_sensor::UsedRamCalculator(ResultPackage* rstptr) {
    uint64_t UsedRam = health_data.TotalRam - health_data.RamAvailable;
    
    const uint64_t GB = 1073741824; // 1024^3
    const uint64_t MB = 1048576;    // 1024^2
    const uint64_t KB = 1024;

    if (UsedRam >= GB) {
        rstptr->Value = (double)UsedRam / GB;
        rstptr->unit = DataUnit::GB;
    } else if (UsedRam >= MB) {
        rstptr->Value = (double)UsedRam / MB;
        rstptr->unit = DataUnit::MB;
    }
}

Concurrency Model

Threading, Synchronization & Lifecycle

Threading Model

Dual-thread architecture for responsiveness:

  • Sampling Thread: OS API calls at configured rate
  • Platform Thread: Flutter EventSink interaction
  • Separation: Minimal work per thread
  • Handoff: Lock-protected queue or concurrent_queue

Communication Patterns

  • Lock-protected shared pointer
  • Encodable payload copy
  • std::deque for queueing
  • Queue draining on platform thread

Shutdown Protocol

  • std::atomic<bool> running flag
  • std::thread join in destructor
  • std::condition_variable for wake
  • Graceful cleanup

Synchronization Rules

  • No locks during heavy ops
  • Value semantics for payloads
  • std::mutex for critical sections
  • Move semantics to minimize copies

Key Code Patterns & Threading Logic

Polling Thread Pseudocode

// Polling thread main loop
while (running) {
    // Sample system state
    sensor.CPU_AttributesRetriver();
    sensor.getsystemtimes();
    
    // Build result package
    EncodableMap payload = build_payload(sensor);
    
    // Push to queue with short lock
    {
        std::lock_guard<std::mutex> lk(queue_mutex);
        queue.push_back(std::move(payload));
    }
    
    // Schedule platform thread task
    platform_task_runner->PostTask([&]{
        drain_queue_and_call_sink();
    });
    
    // Wait before next sample
    std::this_thread::sleep_for(poll_interval);
}
